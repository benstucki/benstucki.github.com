In my previous post we talked about classes and learned most of the building blocks that make them run. I wanted to take a sidebar for a moment to address the importance of events. They aren't an object oriented only concept, but I do believe that understanding events are vitally important to good object oriented programming practices.

Events are implemented in countless and continuously varying ways, but they are all used for the same purpose. Events are the only way for objects to give unsolicited feedback to an application. Let's assume that the Author class used in my previous example has an event called update which is broadcast whenever data is changed by another user. The example below shows how an application might be coded in my own imaginary programming language to respond to this event.

{% highlight javascript %} author = new Author(); author.addEventListener("update", updateApp); private function updateApp( event ) { // update application specific properties } {% endhighlight %}

In this code, we've simply told the application that when the author object fires the update event, it needs to run the updateApp function. This effectively passes control of a part of the application to the author object itself, while still keeping application specific code outside of the reusable and encapsulated class. Notice that the updateApp function also received an event parameter. Most platforms will automatically pass parameters to listener functions as a means for the object to communicate event specific data back to the application. In many frameworks (such as Flex and .NET) commonly used events have already been associated with a predefined function name for you, but that shouldn't keep you from creating your own custom events and event listeners.
